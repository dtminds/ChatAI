# 会话洞察页面设计方案

- 日期：2026-06-01
- 状态：Draft
- 适用范围：AI 客服工作台会话洞察频道页面、导航、关键交互和前端验收
- 关联底座：[2026-06-01-conversation-insights-design.md](./2026-06-01-conversation-insights-design.md)
- 目标入口：`/chat/insights`

## 1. 背景

会话洞察底座已经定义逻辑会话切片、准实时多轮分析、服务质检、实体/意图/风险/行动项、证据链和租户配置。页面设计需要把这些结果组织成用户能直接工作的界面，而不是把所有字段堆在一个洞察页。

第一版页面应围绕客服主管和运营人员的日常任务展开：

```text
今天整体情况怎么样？
哪些问题要优先处理？
服务质检哪里没通过？
哪些业务问题正在升温？
AI 结论有什么证据？
哪些配置需要调整？
```

## 2. 目标与非目标

### 2.1 目标

第一版页面需要支持：

1. 建立会话洞察频道导航，按用户任务拆分页面。
2. 总览页保留“工作台首页”定位，承载全局概览和入口。
3. 服务质检页围绕客户问题是否解决、未解决会话和判定理由组织。
4. 待处理页承接优先处理队列，帮助用户处理风险、跟进和异常。
5. 经营洞察页承接实体/商品、意图、FAQ 机会等业务分析。
6. 分析明细页用于分析结果检索、复核和重新分析。
7. 洞察配置页支持租户配置洞察策略、标签、质检规则和实体词库。
8. 所有页面共享洞察详情和证据消息交互，支持跳转回 `/chat`。

### 2.2 非目标

第一版不做：

- 复杂 BI 大屏。
- 自动绩效扣分或薪酬结算。
- 自动退款、自动发券、自动改订单等交易动作。
- 原始 prompt 编辑和模型参数细调。
- 独立客户画像页。
- 独立商品详情页。
- 移动端完整管理后台体验。

## 3. 页面结构

会话洞察频道建议使用以下路由：

```text
/chat/insights            总览
/chat/insights/quality    服务质检
/chat/insights/follow-ups 待处理
/chat/insights/business   经营洞察
/chat/insights/records    分析明细
/chat/insights/settings   洞察配置
```

第一版优先级：

```text
P0:
- 总览
- 服务质检
- 待处理
- 洞察配置
- 共享详情面板和证据消息

P1:
- 经营洞察
- 分析明细
```

第一版开发只交付 P0 页面。P1 页面可在导航中预留入口，也可后续再接入；如预留路由，只展示空状态或“建设中”，不得影响 P0 页面使用。

导航文案固定为：

```text
总览
服务质检
待处理
经营洞察
分析明细
洞察配置
```

不要使用“会话洞察”作为子页面名。它是整个频道名称，子页面必须表达具体任务。

## 4. 全局布局

会话洞察页面应延续 `/chat` 工作台风格，属于高频操作型页面。视觉风格应克制、密度适中、便于扫描，不做营销式 hero。

通用布局：

```text
频道页头
- 页面标题
- 时间范围
- 分析状态提示
- 配置入口或刷新/重扫入口

二级导航
- 总览 / 服务质检 / 待处理 / 经营洞察 / 分析明细 / 洞察配置

筛选区
- 时间范围
- 会话类型
- 客服席位
- 风险等级
- 意图
- 实体/商品
- 分析状态

主内容区
- 当前页面核心模块

右侧详情面板
- 洞察详情
- 证据消息
- 跳转聊天
- 重新分析
```

前端实现约束：

- 页面放在 `apps/web/src/pages/chat/insights`。
- 页面 API 适配层放在 `apps/web/src/pages/chat/insights/api`。
- 请求通过 `apps/web/src/lib/request.ts`。
- DTO 优先进入 `packages/contracts`。
- UI 优先使用 `apps/web/src/components/ui`。
- 图标使用 Hugeicons，不引入 Lucide。
- 中文短提示类文案默认不加句末标点。

## 5. 总览页

路径：

```text
/chat/insights
```

定位：会话洞察频道首页，回答“今天整体情况怎么样，哪些事情需要优先处理”。

内容固定为：

```text
总览指标
优先处理队列
高风险会话
待跟进事项
意图分布
实体/商品热点
分析完成率和异常状态
```

### 5.1 总览指标

指标建议：

```text
逻辑会话数
已分析会话数
有客户问题会话数
未解决会话数
高风险会话数
待跟进事项数
负面情绪会话数
分析失败/过期数
```

指标卡应支持点击跳转到对应页面或筛选结果：

```text
未解决会话数 -> /chat/insights/quality?resolutionStatus=unresolved
高风险会话数 -> /chat/insights/follow-ups?riskLevel=high
待跟进事项数 -> /chat/insights/follow-ups?actionStatus=open
分析失败/过期数 -> /chat/insights/records?analysisStatus=failed,stale
```

### 5.2 优先处理队列

队列排序信号：

```text
投诉/差评风险
退款/退货待处理
催发货/物流异常
高意向未转化
客服长时间未响应
群聊集中反馈问题
```

每条队列项展示：

```text
风险等级
会话名称
客户或群名称
关联实体
主意图
未解决原因
最后客户消息时间
证据摘要
跳转聊天
```

点击队列项打开共享详情面板。

### 5.3 分布和热点

总览页只展示轻量分布，不做深报表：

```text
意图分布：售前 / 交易 / 物流 / 售后 / 运营机会
实体/商品热点：提及次数、风险会话数、负面反馈数
分析状态：ready / analyzing / failed / stale
```

## 6. 服务质检页

路径：

```text
/chat/insights/quality
```

定位：服务质检和主管辅助复核，围绕“客户是否提出问题，该问题是否解决”组织。

核心问题：

```text
今天有多少逻辑会话？
多少会话客户提出了明确问题？
已解决多少？
未解决多少？
部分解决多少？
未解决的是哪些会话？
未解决理由是什么？
支撑判定的原始消息是什么？
```

### 6.1 质检概览

展示：

```text
逻辑会话总数
已分析会话数
有客户问题会话数
已解决会话数
未解决会话数
部分解决会话数
无明确客户问题会话数
```

状态定义：

```text
resolved：客户问题已得到明确处理或客户表达接受
unresolved：客户问题未被回复、未被覆盖或仍有明确追问/不满
partially_resolved：客服有处理动作，但关键信息、结果或下一步不完整
no_customer_problem：会话中没有明确问题、诉求或投诉
unknown：模型证据不足或分析失败
```

### 6.2 未解决会话列表

字段：

```text
会话名称
客户或群名称
客服席位
客户问题摘要
解决状态
未解决理由
严重等级
证据摘要
最后客户消息时间
分析状态
跳转聊天
```

支持筛选：

```text
时间范围
客服席位
解决状态
严重等级
意图类型
风险类型
分析状态
```

点击列表项打开详情面板，默认定位到 `problem_resolution` 和证据消息。

### 6.3 未解决原因排行

第一版原因枚举：

```text
客服未回复
回复未覆盖客户问题
只安抚但未给处理方案
要求客户等待但未说明下一步
售后/物流/退款进度未确认
客户仍表达不满或继续追问
```

排行项点击后过滤未解决会话列表。

### 6.4 客服维度

字段：

```text
客服席位
处理逻辑会话数
有问题会话数
已解决数
未解决数
部分解决数
未解决率
高严重问题数
平均响应时长
```

第一版文案必须明确：质检结果作为主管辅助复核，不作为自动绩效扣分依据。

## 7. 待处理页

路径：

```text
/chat/insights/follow-ups
```

定位：完整行动队列，回答“现在有哪些事情需要人处理”。

队列类型：

```text
投诉/差评风险
退款/退货待处理
催发货/物流异常
高意向未跟进
长时间未回复
需要主管介入
FAQ/知识库待沉淀
```

列表字段：

```text
优先级
类型
标题
会话名称
客户或群名称
关联实体
主意图
原因
证据摘要
创建时间
最后客户消息时间
状态
```

状态：

```text
open
dismissed
done
expired
```

第一版支持用户将行动项人工标记为 `done` 或 `dismissed`。状态变更只影响洞察模块内的处理状态，不写回平台消息、交易系统或客服绩效。

## 8. 经营洞察页

路径：

```text
/chat/insights/business
```

定位：运营、商品和知识库优化，回答“哪些业务问题值得改进”。

模块：

```text
实体/商品热点
商品负面反馈
售前/交易/物流/售后意图分布
高意向未转化
FAQ 机会
商品说明补充机会
```

实体列表字段：

```text
实体名称
实体类型
提及次数
关联会话数
风险会话数
负面反馈数
购买意向数
未解决问题数
典型问题
证据入口
```

第一版如果没有外部商品、订单、物流主数据，实体全部来自会话分析结果和实体词库。

## 9. 分析明细页

路径：

```text
/chat/insights/records
```

定位：分析结果检索、复核和排障。不要命名为“会话洞察”。

列表字段：

```text
逻辑会话
平台会话 ID
客户或群名称
客服席位
开始时间
结束时间
阶段：live / final
分析状态：ready / analyzing / failed / stale
主意图
情绪
问题解决状态
风险等级
标签
重新分析
```

点击行打开共享详情面板。该页应支持更多技术状态筛选：

```text
analysisStatus
phase
modelProfile
promptVersion
ruleVersion
stale
failed
```

## 10. 洞察配置页

路径：

```text
/chat/insights/settings
```

定位：租户按自身业务调整洞察模块。

配置分组：

```text
洞察策略
自定义标签
质检规则
实体和业务词库
```

第一版配置数据使用种子数据。配置页可先提供读取和展示，完整 CRUD 后续迭代。管理员可见配置页，普通用户不展示配置入口。

### 10.1 洞察策略

洞察策略合并会话结束判定和分析触发配置，避免用户在“切片”和“分析频率”两个技术概念之间来回理解。页面先用业务化预设确定服务节奏，再用少量常用选项微调时机。

控件：

```text
服务节奏：实时客服 / 私域运营 / 自定义
会话空闲多久后视为结束：60 / 120 / 240 分钟
单轮会话最长持续：24 / 48 / 72 小时
会话进行中提前分析：开关
提前分析频率：较快 / 标准 / 较稳
会话结束后多久生成最终结果：立即 / 10 / 20 / 30 分钟
低可信提示阈值：0.1 - 0.9 滑杆
```

不对用户开放新增有效消息数阈值、最小分析间隔、最终分析开关、规则降级开关、迟到消息窗口等底层参数。系统通过“提前分析频率”和“最终结果时机”映射到底层策略，减少配置理解成本。

### 10.2 自定义标签

字段：

```text
标签名称
标签说明
正例
反例
是否启用
是否进入统计
```

### 10.3 质检规则

字段：

```text
规则名称
严重等级
适用场景
判断标准
正例
反例
是否启用
```

第一版规则作为模型判断依据，不作为自动绩效扣分规则。

风险类判断并入质检规则和代办触发，不再作为独立配置分组。

### 10.4 实体和业务词库

字段：

```text
实体类型
标准名称
别名
属性
是否启用
是否参与聚合
```

配置页不提供原始 prompt 编辑、模型参数细调和复杂工作流编排。

## 11. 共享详情面板

所有页面点击会话、风险、质检项、实体、意图、行动项或 FAQ 机会时，使用同一个详情面板。

详情面板结构：

```text
顶部
- 会话名称
- 客户或群名称
- 客服席位
- 分析状态
- 跳转聊天
- 重新分析

摘要区
- 客户诉求
- 处理过程
- 当前结果
- 跟进建议

核心结论
- 问题解决状态
- 风险等级和原因
- 主意图
- 情绪
- 标签
- 实体
- 行动项
- FAQ 机会
- 质检项

证据区
- 证据消息列表
- 证据角色：primary / supporting / context
- 判定理由
- 跳转到 /chat 对应消息
```

证据消息展示要求：

- 按 `msgtime ASC, id ASC` 排序。
- 展示发送方、时间、消息类型和可读内容。
- 高亮支撑当前结论的消息。
- 支持从证据消息跳转 `/chat?conversationId=...&messageId=...`。
- 若证据消息已撤回或不可读，展示状态但保留证据位置。

## 12. 状态设计

页面必须覆盖以下状态：

```text
loading
empty
ready
analyzing
failed
stale
partial
permission denied
```

状态文案建议：

```text
分析中：语义洞察生成中
失败：语义洞察生成失败
过期：有新消息，洞察结果待更新
部分完成：部分维度暂未生成
空状态：当前筛选条件下暂无洞察
```

模型失败时页面仍展示基础 SQL 和规则降级结果，不应整页空白。

## 13. API 映射

页面 API 建议：

```text
GET /api/server/insights/summary
GET /api/server/insights/quality
GET /api/server/insights/follow-ups
GET /api/server/insights/business
GET /api/server/insights/records
GET /api/server/insights/sessions/:sessionId
POST /api/server/insights/sessions/:sessionId/reanalyze
```

配置 API 使用底座 spec 中定义的配置接口。

详情接口必须返回：

```text
logical session metadata
current snapshot
summary
problem resolution
qa findings
sentiment
tags
entities
intents
risks
action items
faq candidates
evidence message contexts
analysis status
```

## 14. 权限与可见性

第一版数据页不做额外角色权限，但必须遵守登录态、租户和会话数据隔离：

- 用户只能看到当前登录态和租户范围内可见的账号、席位、会话和客户。
- 证据消息回查也必须做同样权限校验。
- 配置页只对管理员展示。
- 普通用户可查看数据页，但不应看到配置入口。

## 15. 响应式策略

桌面端优先。第一版移动端只保证可读和可操作：

- 总览指标改为纵向列表或双列。
- 表格可切换为列表卡片。
- 详情面板在窄屏变为全屏抽屉。
- 筛选区收进筛选面板。
- 不隐藏跳转聊天和证据查看入口。

## 16. 测试策略

Contracts：

- 覆盖 summary、quality、follow-ups、business、records、detail DTO。
- 校验证据消息上下文必填字段。

Backend：

- 筛选参数测试。
- 权限过滤测试。
- 质检统计测试。
- 未解决会话列表测试。
- 证据上下文查询和排序测试。
- 重新分析任务创建测试。

Web：

- 总览页指标和跳转。
- 服务质检页解决状态筛选。
- 待处理页状态切换。
- 详情面板展示证据消息。
- 跳转 `/chat` 参数正确。
- 配置页表单校验。
- loading、empty、failed、stale、partial 状态。

测试不应断言 Tailwind class、字号、间距、阴影、圆角等易变视觉实现细节。

## 17. 第一版验收标准

第一版完成时应满足：

1. `/chat/insights` 能展示总览指标、优先处理队列、高风险会话、待跟进事项、意图分布、实体/商品热点、分析完成率和异常状态。
2. `/chat/insights/quality` 能统计逻辑会话、客户问题会话、已解决、未解决、部分解决，并列出未解决会话和判定理由。
3. `/chat/insights/follow-ups` 能展示风险、跟进和异常行动队列，并支持按类型、优先级和状态筛选。
4. `/chat/insights/follow-ups` 支持将行动项人工标记为 `done` 或 `dismissed`。
5. `/chat/insights/settings` 管理员可见，并能展示种子数据提供的洞察策略、标签、质检规则和实体词库。
6. 共享详情面板能展示摘要、问题解决状态、风险、质检、标签、实体、意图、行动项、FAQ 和分析状态。
7. 任意结论都能展示证据消息上下文，并跳转回 `/chat` 对应会话和消息位置。
8. 分析中、失败、过期、部分完成和空状态都有明确展示。
9. 模型失败时页面仍展示基础统计和规则降级结果。
10. 普通用户和管理员看到的配置入口符合权限。
11. P1 页面 `/chat/insights/business` 和 `/chat/insights/records` 后置；如预留路由和导航空状态，不能影响 P0 页面使用。
