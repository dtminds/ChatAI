# 洞察配置数量限制 — 需求文档

## 背景

当前系统中，意图配置、标签体系、质检规则、实体词库四类配置记录在数据库层面**没有数量上限**。用户可以无限制创建配置，但送入 LLM 分析时会被 `slice(0, N)` 静默截断——超出部分直接丢弃，既无报错也无提示。

这带来两个问题：
1. LLM 分析精度随配置数量增加而下降（选项越多越容易混淆）
2. 用户不知道自己的配置是否全部参与了分析，可能误以为 100 条意图都在生效

## 设计目标

- 为每类配置引入**已启用数**的硬上限，控制 LLM 输入规模
- 为每类配置引入 50 条未删除记录总数上限，避免无分页列表无限膨胀
- 超限时给用户清晰的交互反馈，杜绝静默丢弃
- 上限值由后端统一下发，前后端和 worker 使用同一套口径

## 配置上限

| 配置类型 | 已启用硬上限 | 软提示阈值 |
|---------|------------|-----------|
| 意图配置（intentConfigs） | 20 | 15 |
| 标签体系（labelConfigs） | 20 | 15 |
| 质检规则（qaRuleConfigs） | 10 | 8 |
| 实体词库（entityDictionary） | 20 | 15 |

以上为**已启用（enabled = true）** 的记录数上限，不是记录总数上限。

每类配置另有**未删除记录总数上限 50 条**：
- 统计范围为 `status != -1` 的记录，包含启用和停用
- 删除后的记录不计入总数额度
- 总数上限只在新建时校验；编辑、停用、启用、删除不受总数上限影响

## 适用前提

- 本需求默认在正式上线前落地，不考虑存量“已启用数已超上限”的历史数据治理
- 上线后的正常路径中，不允许再产生新的超限启用状态

## 核心规则

### 1. 启用拦截

当某类配置的已启用数量达到硬上限时：
- 禁止将新的记录从停用切换为启用
- 新建记录时，如果默认为启用状态且已达上限，后端拒绝创建并返回明确错误
- 编辑记录时，只要保存后的 `status = 1`，同样必须校验是否仍允许启用
- 返回的错误信息需指明：当前已启用 N 条，上限为 M 条，请先停用后再操作

所有会导致记录最终保存为启用状态的写路径都必须执行同一套校验，包括但不限于：
- 新建 `POST`
- 编辑保存 `PUT`
- 单独切换状态 `PATCH /status`

### 2. 停用不受限制

- 停用操作始终允许，不受任何限制
- 已停用的记录不占用启用配额

### 3. 新建总数限制

- 每类配置最多保留 50 条未删除记录
- 新建时如果该类配置 `status != -1` 的记录数已达到 50，后端拒绝创建
- 新建时如果未达总数上限，但保存为启用状态，仍需继续校验启用上限
- 编辑、停用、启用和删除不校验总数上限

### 4. LLM 送入策略

送入 LLM 的配置仅包含已启用的记录。在已启用记录中：
- **意图配置**：按 `weight` 升序排序（优先级高的在前）
- **质检规则**：按 `severity` 排序（high > medium > low）
- **标签体系**：按 `includeInStatistics` 优先，再按创建时间倒序
- **实体词库**：按 `includeInAggregation` 优先，再按创建时间倒序

LLM 输入上限与启用上限保持一致：
- 意图配置最多送入 20 条
- 标签体系最多送入 20 条
- 质检规则最多送入 10 条
- 实体词库最多送入 20 条

正常路径下，不再保留旧的“大数量 slice 截断”逻辑，也不存在“静默截断后再提示 LLM”的方案。

### 5. 列表展示策略

- 设置页列表排序与 LLM 送入排序解耦
- 四类配置列表统一按 `id` 倒序展示
- 顶部 summary 展示当前已启用数 / 上限，不在各表格标题区重复展示

### 6. 错误契约

后端发现启用超限时，必须返回稳定错误码，供前端做强提醒处理。

建议错误结构：

```json
{
  "success": false,
  "error": {
    "code": "INSIGHT_CONFIG_ENABLED_LIMIT_REACHED",
    "message": "当前已启用 N 条（上限 M 条），请先停用其他配置",
    "details": {
      "configType": "intentConfigs",
      "currentEnabled": N,
      "limit": M
    }
  }
}
```

其中：
- `code` 为稳定错误码，前端不得依赖 message 文案做分支
- `configType` 取值与设置接口字段保持一致：`intentConfigs`、`labelConfigs`、`qaRuleConfigs`、`entityDictionary`
- `currentEnabled` 表示当前已启用数量
- `limit` 表示该类配置的硬上限

后端发现新建超过总数上限时，也必须返回稳定错误码：

```json
{
  "success": false,
  "error": {
    "code": "INSIGHT_CONFIG_TOTAL_LIMIT_REACHED",
    "message": "当前已有 N 条配置（上限 50 条），请先删除无用配置后再新建",
    "details": {
      "configType": "intentConfigs",
      "currentTotal": N,
      "limit": 50
    }
  }
}
```

其中：
- `currentTotal` 表示该类当前未删除记录数
- `limit` 固定为 50

### 7. 前端交互

#### 顶部 summary 计数器

顶部 summary 展示每类配置的当前状态：

```
意图配置     15/20   [+ 新建意图]
```

- **< 软提示阈值**：正常显示计数
- **>= 软提示阈值，< 硬上限**：黄色/橙色提示「配置较多，可能影响分析精度」
- **= 硬上限**：仍允许用户尝试启用或保存启用状态，但后端会拒绝并触发强提醒

summary 所需的上限值由后端下发，前端不硬编码。

#### 启用/停用交互

- 用户尝试启用一条已达上限的配置时，允许点击开关
- 后端拒绝后，前端将开关回滚到原状态
- 同时弹出**强提醒弹窗**，提示：「当前已启用 N 条（上限 M 条），请先停用其他配置」
- 停用操作无任何限制
- 启用超限场景不使用 toast 提示

#### 新建配置交互

- 新建对话框中，「启用」开关默认开启
- 如果当前已启用数量达到上限，「启用」开关默认关闭，并提示用户「已达到上限，创建后需手动启用」
- 如果用户主动将「启用」开关打开并提交保存，后端仍需做最终校验
- 如果新建/编辑保存时发现超限，前端保留当前表单内容，并弹出**强提醒弹窗**，不使用 toast
- 如果新建时发现总数超限，前端同样保留当前表单内容，并弹出**强提醒弹窗**，不使用 toast

## 不涉及的范围

- 不改变 LLM 分析的核心逻辑（仅确保启用数量受控）
- 不改动配置的字段结构
